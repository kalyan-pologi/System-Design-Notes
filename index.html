⚙️ Patterns + Rules

Kafka Deep Dive

Scalability
Hot partitions → remove key / compound key / back pressure
Fault tolerance
1 leader + 3 follower replicas
acks=all → guaranteed no data loss
Consumer commits offset only after work done
Rebalance consumer group on failure
Error & retries
Producer retries on failure
Consumer → retry topic → DLQ
Performance
Batch processing + compression at producer
Retention policy
7 days time-based · 1 GB size-based purging

Sharded Counters + Write Rules

Problem
Single counter = write bottleneck (lock contention)
Solution
Create N shards, pick random shard per write
Read = sum all N shards
Contention reduced by 1/N
Used in: likes, views, inventory counts
Write scaling rules
Writes 40× more expensive than reads
Global shared data = lock contention = slow
Architect for parallel writes, not serial
Compress before sending over the network
Avoid disk seeks — keep hot data in memory

Interview Tips

Round numbers — 99,987 → 100,000
Label units — write "5 MB" not just "5"
State assumptions first, then calculate
Process > precision — show thinking aloud
Peak = 2× avg as safe default
Practice: QPS, storage, cache, # servers
Latency feel rules
<10 ms → real-time (gaming, HFT)
<100 ms → snappy (web, chat apps)
~1 sec → feels slow or broken
Quick estimates
Servers: 50K RPS ÷ 5K/server = 10 servers
Storage: users × items/day × size × days

🎯 Delivery Framework

How to structure a 45–60 min system design interview. Follow this sequence to stay focused, avoid rabbit holes, and always deliver a working system.

1️⃣ Requirements (~5 min)

Functional Requirements

Ask these questions
What are the core use cases? (top 3 only)
Who will use the system?
How many users? (scale context)
What usage patterns? (read-heavy / write-heavy / bursty?)
What is explicitly out of scope?
What does MVP look like?
Twitter Example
Users can post tweets
Users can follow other users
Users see a feed of followed users
⚠️ Keep it to top 3. Long requirement lists hurt more than they help.

Non-Functional Requirements — SCALE-FCD

S — Scalability
Read/write ratio, bursty traffic, holiday spikes? Apply: LB, sharding, CQRS, CDN, autoscaling
C — CAP Theorem
Consistency vs availability? Apply: leader-follower, quorum, Raft/Paxos, eventual consistency
A — Availability
What uptime SLA? 99.9% / 99.99%? Apply: redundancy, failover, health checks, replicas
L — Latency
How fast? Which ops matter most? Apply: caching, async, indexing, CDN, prefetching
E — Environment
Mobile? Low bandwidth? Offline? Apply: compression, delta sync, adaptive bitrate, edge cache
F — Fault Tolerance
Handle failures? Apply: retries, backoff, circuit breaker, chaos testing, bulkheading
C — Compliance
GDPR, HIPAA, data locality? Apply: anonymize, audit logs, soft delete, retention policy
D — Durability
Can we afford data loss? Apply: WAL, multi-region replication, backups, S3/GCS
⚠️ Always quantify. "Low latency" is vague. "Feed loads in <200ms" is useful. Pick top 3–5 NFRs.

Capacity Estimation — Only if it changes design

Traffic
QPS formulaDAU × actions ÷ 86,400
Peak QPS2–10× avg
Read/Write ratioestimate separately
Storage
Formula: size × items/user × users × retention days
Separate hot data vs cold data
Separate media vs metadata
Memory (Cache)
Cache targettop 20% hot data
RAM neededhot data × 1.2×
Servers
Formula: total QPS ÷ QPS per server
1 CPU core ≈ 1K lightweight req/s
✅ Tip: Skip estimation if conclusion is just "it's a lot." Only calculate when it directly changes your design choices.
2️⃣ Core Entities (~2 min)    3️⃣ API Design (~5 min)

Core Entities

List the nouns your system revolves around
These become your API resources + DB tables
Don't list every field yet — just entity names
Ask: who are the actors? what are the resources?
Good names matter — avoid generic placeholders
Twitter Example
Useractor
Tweetcore resource
Followrelationship
Feedderived resource

API Design

Protocol — pick one
REST — default for most interviews. CRUD on resources
GraphQL — diverse clients, flexible data needs
gRPC / RPC — internal APIs, high performance, service-to-service
WebSockets — real-time, bidirectional (chat, live feed)
SSE — server push, one-directional (notifications)
REST Rules
Use plural nouns as resource names
Derive user from auth token, not request body
Version your API: /v1/tweets
Never expose sensitive IDs from user input
Twitter REST Example
POST /v1/tweets
   body: { text: string }

GET    /v1/tweets/{tweetId}

POST /v1/follows
   body: { followee_id: string }

GET    /v1/feed → Tweet[]

DELETE /v1/follows/{userId}
4️⃣ Data Flow (optional, ~5 min)    5️⃣ High Level Design (~15 min)

Data Flow (Optional)

Skip if system has no long processing pipeline
Use for: crawler, recommendation engine, ETL, stream processing
Just an ordered list — inputs → transforms → outputs
Web Crawler Example
1. Fetch seed URLs
↓
2. Parse HTML
↓
3. Extract URLs
↓
4. Store data
↓
5. Repeat

High Level Design

Approach
Go one API endpoint at a time — build sequentially
Draw boxes + arrows: client → LB → server → DB
Talk through data flow for every request
Note DB schema columns next to the DB box
Call out complexity areas verbally — don't fix yet
Satisfy functional requirements first, NFRs in deep dive
⚠️ #1 mistake: Adding too much complexity too early and never finishing. A complete simple design beats an incomplete brilliant one.

6️⃣ Deep Dive (~10–15 min)

Scaling

Horizontal Scaling
Add servers behind a load balancer
Stateless servers — no session on server
Autoscaling — handle bursty / holiday traffic
Load Balancer Types
Active-Passivefailover standby
Active-Activeboth serve traffic
Layer 4 (TCP)fast, no content inspect
Layer 7 (HTTP)route by URL/header
App Layer Scaling
Microservices — scale each service independently
Service Discovery — services find each other dynamically
CQRS — separate read and write models
Read replicas — offload reads from primary DB

Database Deep Dive

SQL — RDBMS
Master-Slave — one write, many reads
Master-Master — multi write, complex conflicts
Federation — split by function (users DB, products DB)
Sharding — split by key across nodes
Denormalization — duplicate data to avoid joins
Indexes — faster reads, slower writes (+10–30%)
NoSQL Types
Key-ValueRedis, DynamoDB
Wide-ColumnCassandra, HBase
DocumentMongoDB, Couchbase
GraphNeo4j, Amazon Neptune
Time SeriesInfluxDB, Prometheus
VectorPinecone, pgvector
Fast Lookup Guide
Bounded size (RAM)Redis, Memcached
AP unboundedCassandra, DynamoDB
CP unboundedHBase, MongoDB
DB Partitioning Methods
Horizontalrows across nodes
Verticalcolumns across nodes
Directory-basedlookup table
Range-basedA–M node1, N–Z node2
Hash-basedconsistent hashing
Round Robindistribute evenly

Caching Deep Dive

Cache Levels (client → DB)
Client-side — browser cache, local storage
CDN — edge cache near user
API Gateway / Webserver — request level
Application — in-memory, object level
Database — query result cache
Cache Strategies
Cache-asideapp manages cache
Write-throughwrite cache + DB together
Write-behindwrite cache, async DB
Refresh-aheadpre-populate before expiry
Eviction Policies
LRUleast recently used
LFUleast frequently used
FIFOfirst in first out
Target: 80–90% hit rate. Top 20% data = 80% of benefit.
Redis hit latency: ~1ms vs DB 20–50ms

Communication Protocols

Transport Layer
TCPreliable, ordered, slower
UDPfast, no guarantee (video/gaming)
Application Layer
HTTP/RESTdefault, stateless, simple
gRPC / RPCinternal, fast, protobuf
WebSocketsreal-time, bidirectional
SSEserver push, one-way stream
Long pollingnear real-time, simpler
Protocol Numbers
gRPC vs REST payload~7× smaller
Protobuf vs JSON3–10× smaller
HTTP/2 multiplexing1 conn, N streams
WebSocket overhead~2B per frame
Proxy Types
Reverse Proxy — sits in front of servers (Nginx). Handles SSL, compression, caching, LB
API Gateway — reverse proxy + auth, rate limiting, routing, monitoring

Asynchronism + Reliability

Async Patterns
Message Queues — decouple producer/consumer (Kafka, RabbitMQ, SQS)
Task Queues — background jobs, heavy processing (Celery)
Back Pressure — slow producer when consumer is overwhelmed
Event Sourcing — store events not state
Reliability Patterns
Retries — retry on failure with limits
Exponential Backoff — wait longer between retries
Circuit Breaker — stop calls to failing service
Bulkheading — isolate failure domains
Failover — active-passive switch on failure
Chaos Testing — inject failures proactively
Replication
Primary-Secondaryone write, sync reads
Primary-Primarymulti write, conflict risk
Same region lag~10 ms
Cross region lag50–150 ms

Durability + Security

Durability
Write-ahead logs (WAL) — no data loss on crash
Multi-region replication — survive datacenter failure
RAID — disk-level redundancy
Incremental + full backups — point-in-time recovery
Durable cloud storage — S3 (11 nines durability)
Consensus-based storage — CockroachDB, Google Spanner
Security Checklist
HTTPS + TLS — encryption in transit
Encryption at rest — AES-256
Auth — OAuth 2.0 / JWT tokens
RBAC / ABAC — role and attribute-based access
Firewall — limit egress + ingress
Principle of least privilege — minimal permissions
Input validation — prevent injection attacks
Rate limiting — prevent DDoS / abuse

Justify Your Design (~3 min)

Walk through throughput of each layer
Explain latency between each layer
Show overall latency meets NFR targets
Confirm each NFR is addressed in design
Call out trade-offs made and why
Acknowledge what you'd improve with more time
Example Justification
"Feed read latency target is <200ms. With Redis cache hit (~1ms) for 80% of requests, and DB reads (~5ms) for the rest via read replica, p99 is well under target."

Interview Tips

Talk while you draw — silence makes interviewers nervous
Note complexity, don't fix it yet — "I'll revisit in deep dive"
Give interviewer space — they have specific signals to probe
Senior = proactive deep dives · Junior = respond to prompts
Never skip to deep dive before finishing high level design
Finish > perfect — complete simple design beats incomplete brilliant one
Ask clarifying questions like talking to a PM, not an examiner
Seniority Guide
Juniorprompted → responds well
Mid-leveldelivers working system
Seniorproactive, drives trade-offs
Staffleads, challenges requirements
