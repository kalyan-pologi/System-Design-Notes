⚙️ Patterns + Rules

Kafka Deep Dive

Scalability
Hot partitions → remove key / compound key / back pressure
Fault tolerance
1 leader + 3 follower replicas
acks=all → guaranteed no data loss
Consumer commits offset only after work done
Rebalance consumer group on failure
Error & retries
Producer retries on failure
Consumer → retry topic → DLQ
Performance
Batch processing + compression at producer
Retention policy
7 days time-based · 1 GB size-based purging

Sharded Counters + Write Rules

Problem
Single counter = write bottleneck (lock contention)
Solution
Create N shards, pick random shard per write
Read = sum all N shards
Contention reduced by 1/N
Used in: likes, views, inventory counts
Write scaling rules
Writes 40× more expensive than reads
Global shared data = lock contention = slow
Architect for parallel writes, not serial
Compress before sending over the network
Avoid disk seeks — keep hot data in memory

1️⃣ Requirements (~5 min)

Functional Requirements

Ask these questions
What are the core use cases? (top 3 only)
Who will use the system?
How many users? (scale context)
What usage patterns? (read-heavy / write-heavy / bursty?)
What is explicitly out of scope?
What does MVP look like?
Twitter Example
Users can post tweets
Users can follow other users
Users see a feed of followed users
⚠️ Keep it to top 3. Long requirement lists hurt more than they help.

Non-Functional Requirements — SCALE-FCD

S — Scalability
Read/write ratio, bursty traffic, holiday spikes? Apply: LB, sharding, CQRS, CDN, autoscaling
C — CAP Theorem
Consistency vs availability? Apply: leader-follower, quorum, Raft/Paxos, eventual consistency
A — Availability
What uptime SLA? 99.9% / 99.99%? Apply: redundancy, failover, health checks, replicas
L — Latency
How fast? Which ops matter most? Apply: caching, async, indexing, CDN, prefetching
E — Environment
Mobile? Low bandwidth? Offline? Apply: compression, delta sync, adaptive bitrate, edge cache
F — Fault Tolerance
Handle failures? Apply: retries, backoff, circuit breaker, chaos testing, bulkheading
C — Compliance
GDPR, HIPAA, data locality? Apply: anonymize, audit logs, soft delete, retention policy
D — Durability
Can we afford data loss? Apply: WAL, multi-region replication, backups, S3/GCS
⚠️ Always quantify. "Low latency" is vague. "Feed loads in <200ms" is useful. Pick top 3–5 NFRs.

API Design

Protocol — pick one
REST — default for most interviews. CRUD on resources
GraphQL — diverse clients, flexible data needs
gRPC / RPC — internal APIs, high performance, service-to-service
WebSockets — real-time, bidirectional (chat, live feed)
SSE — server push, one-directional (notifications)
REST Rules
Use plural nouns as resource names
Derive user from auth token, not request body
Version your API: /v1/tweets
Never expose sensitive IDs from user input

⚡ Golden Rules — If X then Y

Scaling Rules

If traffic is high
→ Load balancer + horizontal scaling
→ Autoscaling for bursty/unpredictable traffic
→ Stateless servers — no session on server side
If read-heavy
→ Read replicas to offload primary DB
→ Cache aggressively (Redis, CDN, app-level)
→ Denormalize DB to avoid expensive joins
If write-heavy
→ Message queue (Kafka/SQS) to absorb spikes
→ Batch writes to reduce DB pressure
→ Sharding to distribute write load
→ Write-behind cache for async persistence
If data is huge
→ Shard by user ID / region / hash
→ Archive cold data to cheap storage (S3 Glacier)
→ Columnar DB (Cassandra) for ever-growing data

Technology Selection Rules

If you need search
→ Elasticsearch for full-text / fuzzy search
→ Trie for autocomplete / prefix search
If you need fast lookups
→ Redis for bounded, in-memory (sessions, cache)
→ Cassandra for unbounded AP data
→ HBase / MongoDB for unbounded CP data
If you need real-time
→ WebSockets for bidirectional (chat, gaming)
→ SSE for server push (notifications, feeds)
→ Kafka for event streaming at scale
If you need large file storage
→ S3 + CDN — never store blobs in DB
→ Multipart upload for files >100MB
If you need global low latency
→ CDN for static content at edge
→ Multi-region deployment for dynamic content
→ GeoDNS to route users to nearest region

Reliability + Design Rules

If a service can fail
→ Circuit breaker to stop cascading failures
→ Retry + exponential backoff with jitter
→ Fallback / graceful degradation — serve stale data
→ Bulkhead — isolate failure domains
If data must not be lost
→ WAL (write-ahead log) before committing
→ acks=all in Kafka — wait for all replicas
→ Multi-region replication + regular backups
If operations must be safe to retry
→ Make APIs idempotent (same result on retry)
→ Use idempotency keys for payments/writes
If deploying new features
→ Feature flags — enable/disable without deploy
→ Canary deploy — 1% traffic first, then ramp
→ Blue-green deploy — instant rollback
If handling overload
→ Rate limiting — token bucket / sliding window
→ Back pressure — slow producer when consumer overwhelmed
→ Load shedding — drop low-priority requests
🧠 Self-Check Questions — Ask During Interview

Requirements + Scale

Who is the customer and what is their top priority?
What is the minimum viable feature set?
What is the most critical NFR — performance, availability, or consistency?
What are the SLAs — latency, uptime, durability?
What is the expected scale today and in 1–2 years?
Where will the system grow or change most frequently?
Are there regulatory constraints — GDPR, HIPAA, data locality?
How will you handle traffic spikes — flash sales, virality?
💡 Ask these like a PM, not an examiner. Shows you think about real impact.

Design + Trade-offs

Where are the bottlenecks and single points of failure?
What CAP trade-off are you making — consistency or availability?
What caching strategies reduce load and latency?
Are you optimizing for storage cost vs retrieval speed?
What parts need horizontal scaling and how is it achieved?
If a service fails, how will it degrade gracefully?
Can the system auto-scale based on load?
Are there clear contracts / interfaces between components?
How will dependencies (3rd party APIs, DBs) impact your system?
💡 Every design has trade-offs. Call them out proactively — don't wait to be asked.

Operations + Security

What metrics and alerts will you monitor?
What logs and traces help with root cause analysis?
How will you support zero-downtime deployments?
What are the backup and recovery strategies for data loss?
Are operations designed with idempotency + retries in mind?
How is user data secured — encryption at rest and in transit?
Are PII / sensitive data excluded from logs?
How will incident response + rollback work during outage?
Can you run load tests + chaos testing to validate assumptions?
Are you building in observability from the start — not after?
💡 Senior candidates raise operational concerns unprompted. This is what separates mid from senior.
