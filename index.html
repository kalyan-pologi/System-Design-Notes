⚙️ Patterns + Rules

Kafka Deep Dive

Scalability
Hot partitions → remove key / compound key / back pressure
Fault tolerance
1 leader + 3 follower replicas
acks=all → guaranteed no data loss
Consumer commits offset only after work done
Rebalance consumer group on failure
Error & retries
Producer retries on failure
Consumer → retry topic → DLQ
Performance
Batch processing + compression at producer
Retention policy
7 days time-based · 1 GB size-based purging

Sharded Counters + Write Rules

Problem
Single counter = write bottleneck (lock contention)
Solution
Create N shards, pick random shard per write
Read = sum all N shards
Contention reduced by 1/N
Used in: likes, views, inventory counts
Write scaling rules
Writes 40× more expensive than reads
Global shared data = lock contention = slow
Architect for parallel writes, not serial
Compress before sending over the network
Avoid disk seeks — keep hot data in memory

Interview Tips

Round numbers — 99,987 → 100,000
Label units — write "5 MB" not just "5"
State assumptions first, then calculate
Process > precision — show thinking aloud
Peak = 2× avg as safe default
Practice: QPS, storage, cache, # servers

<10 ms → real-time (gaming, HFT)
<100 ms → snappy (web, chat apps)
~1 sec → feels slow or broken
Quick estimates

Storage: users × items/day × size × days
